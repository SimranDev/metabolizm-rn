@AGENTS.md

## Product

**Metabolizm** is a Health & Fitness app (iOS + Android) — a "swiss-knife for fitness" whose **core is weight, calorie, and macro/micro tracking**, with many surrounding features. Users create a profile, set goals (gain muscle / lose weight / maintain, with BMR·TDEE-based targets), log food by search or **barcode scan**, get **recipe** suggestions, view a **health dashboard**, and import **steps/activity** from other fitness apps.

Intended stack for feature work:
- **Backend:** cloud from the start — auth + hosted database + multi-device sync.
- **Activity/steps:** Apple Health (iOS) + Android **Health Connect**.
- **Food data / barcodes:** **Open Food Facts + USDA** FoodData Central.
- **Monetization:** subscription tiers (free / pro / pro max) — the header's plan icon reflects the tier.

## Priorities & constraints

**Performance and small app size come first — ahead of animations and heavy assets.** Prefer lightweight dependencies and native primitives (SF Symbols on iOS, Material `md` symbols on Android) over large bundled images or Lottie; lazy-load where sensible and watch bundle/app size as features land.

## Project

pnpm monorepo (workspaces: `apps/*`, `packages/*`; single lockfile at root, default isolated linking):

- [apps/mobile/](apps/mobile/) — the Expo SDK 57 app (React Native 0.86, React 19.2, expo-router v57), package name `mobile`, targeting **iOS and Android only — there is no web version**. Some `*.web.tsx` split files remain from the Expo starter but are not a shipping target. TypeScript strict mode.
- [apps/api/](apps/api/) — the backend, package name `api`: NestJS 11 + Fastify 5 (tsc builder — **do not add `incremental` to its tsconfig**: `nest build` with TS 6 + incremental silently emits nothing, nest-cli#3312), global prefix `/v1`, fail-fast env validation via `@nestjs/config` + zod ([src/config/env.ts](apps/api/src/config/env.ts)). Drizzle ORM + postgres-js behind a global DI module ([src/db/db.module.ts](apps/api/src/db/db.module.ts), inject with the `DB` token); the schema itself lives in `@metabolizm/db`. Domain modules auth/users/catalog/diary/sync/groups/summaries/weight are implemented; billing is an empty skeleton. `summaries` is both the internal write-side service every diary/weight/target write recomputes through *and* a read controller (`GET /v1/summaries/days`) serving the caller's own rollups — only `SummariesService` is exported to other modules. `users` owns `GET`/`PATCH`/`DELETE /v1/users/me` and is the **only writer of `users.timezone`**, which every server-side "today" pivots on — it defaults to `'UTC'`, so a client that never PATCHes it has entry dates and streaks silently shifted by its real offset. `DELETE /v1/users/me` is account deletion: one `DELETE FROM users` does nearly all of it, since every table referencing `users.id` cascades (including Better Auth's `sessions`/`accounts`, which is what kills every device's session). The one exception is `groups.owner_id`, deliberately `ON DELETE RESTRICT` so an owner can never silently orphan a group — `GroupsService.releaseOwnedGroups` runs first **in the same transaction**, transferring each owned group to its most senior remaining active member (oldest membership breaking the tie) and hard-deleting only the ones nobody else is left in. Hard, not the usual soft delete: a `deleted_at` row still holds the FK. `diary_entries.food_id` and `user_targets.set_by` are `SET NULL`, so another member's logged history and a coach-set target survive the setter's departure. This is the only reason `UsersModule` imports `GroupsModule`. No Prettier (repo-wide convention); tests are the groups masking and weight math suites (see Commands).
- [packages/db/](packages/db/) — `@metabolizm/db`: the Drizzle schema ([src/schema.ts](packages/db/src/schema.ts)), committed migrations in `packages/db/drizzle/`, the drizzle-kit config, and a prod migration runner compiled to `dist/migrate.js` (shipped inside the api Docker image; run with `node node_modules/@metabolizm/db/dist/migrate.js`). Node-only, consumed by api + admin: `main`/`types` → `dist/` built by tsc (`prepare` rebuilds on install), deliberately **no** `react-native` condition — never import it from mobile. The `db:generate`/`db:migrate`/`db:studio`/`db:dbml` root scripts proxy here; [drizzle.config.ts](packages/db/drizzle.config.ts) loads `apps/api/.env` (the single source of truth for `DATABASE_URL`). Also holds the committed DBML schema export `schema.dbml` and the dev-only script that writes it ([scripts/generate-dbml.ts](packages/db/scripts/generate-dbml.ts)) — both `scripts/` and `drizzle.config.ts` sit outside the build tsconfig (`include: ["src"]`), so they never reach `dist/`.
- [apps/admin/](apps/admin/) — **internal, dev-only food-catalog admin tool; never deployed.** Package name `admin`, run via `pnpm admin`: plain Fastify 5 server ([server/](apps/admin/server/), tsx watch on :4000, own zod-validated `.env`) + Vite React 19 SPA ([web/](apps/admin/web/), :5173, proxies `/api`). `POST /api/parse` deterministically maps a pasted USDA FoodData Central food JSON into a per-100 food record via the shared mapper in [server/usda-mapper.ts](apps/admin/server/usda-mapper.ts) — the same module `scripts/import-usda.ts` uses to bulk-import Foundation/SR Legacy download files (idempotent upsert on `foods.source_ref`, run via `pnpm --filter admin import:usda`); `/api/foods` CRUD writes **system** catalog rows (`ownerId: null`, `source: "system"`, public, verified, UUIDv7) directly with Drizzle — deliberately not via the api's user-scoped `/v1/catalog`. It intentionally has **no `build` script** so the Dockerfile's `pnpm run -r build` never builds it.
- [packages/shared/](packages/shared/) — `@metabolizm/shared`: pure data shapes + API contract types shared between app and backend (catalog, health, diary, groups, profile, api), plus the canonical nutrient registry ([src/nutrients.ts](packages/shared/src/nutrients.ts)) and the catalog food-input zod schemas ([src/catalog-schemas.ts](packages/shared/src/catalog-schemas.ts), used by both api and admin). Dual-consumed via conditional exports — **`react-native` condition (must stay first; conditions match in key order) → `src/index.ts`** so Metro and the mobile tsc use live TS source with no build step, while `types`/`default` → `dist/` (CJS + d.ts, built by `tsc`; the `prepare` script rebuilds it on every `pnpm install`) for Node/api. After editing shared, rebuild for the api with `pnpm --filter @metabolizm/shared build` (the `pnpm api` script does this on start). **Rule:** pure types → `packages/shared`; runtime helpers (calc, unit conversions, portion math, the catalog API client) → `apps/mobile/src/lib`. Never add `react`/`react-native` to its dependencies.

Metro is auto-configured for the workspace by `expo/metro-config` — there is deliberately **no metro.config.js**; don't add one.

Expo APIs changed significantly in SDK 57 — consult https://docs.expo.dev/versions/v57.0.0/ before writing Expo-related code rather than relying on prior knowledge.

## Groups — the masking invariant

Social accountability: private, invite-only groups (`partner` | `family` | `friends` | `trainer`, ≤ ~50 people) whose members share nutrition data with each other. No public discovery, no follower graph, and deliberately **no feed table** — feeds are computed at read time (fan-out-on-read).

Each membership carries its own `share_config` (`calories`, `macros`, `mealNames`, `mealDetail`, `weightTrend`, `streaks`, plus `adherenceOnly`); the category picks the defaults at join time ([CATEGORY_SHARE_DEFAULTS](packages/shared/src/groups-schemas.ts)) and the member can edit them per group at any time.

**Non-negotiable, and the reason [groups/masking.ts](apps/api/src/groups/masking.ts) and its regression suite ([masking.test.ts](apps/api/src/groups/masking.test.ts), `pnpm test`) exist:** all masking happens server-side in the service layer before serialization. Payloads are built **allowlist-style** — a field is *added* only when the subject's config exposes it, never copied wholesale and stripped — so an unshared field is **absent from the JSON, not null**, including in nested/joined data. Anything new that reads another member's data goes through `masking.ts`; never serialize a `daily_summaries` or `diary_entries` row into a group response directly.

Rules that fall out of that:
- `adherenceOnly` returns server-computed booleans (`proteinHit`, `caloriesInRange`, `logged`) — absolute kcal/gram/weight numbers never leave the API for that member.
- Masking reads the member's **current** config at read time, so a change takes effect immediately for all past and future days. There are no consent snapshots.
- `status` `left`/`removed` excludes that member from every group read (loads are scoped to `active` members, so their rows never enter a query).
- Leaderboards rank by adherence % and logging streak only — **never** by raw calories or weight, which aren't in the response at all.
- Share-config *updates* are partial patches ([groupSharePatchSchema](packages/shared/src/groups-schemas.ts)) with no per-field defaults. A schema with `.default(false)` per field would materialize every unmentioned toggle as `false` and silently switch off what the caller didn't name; `groupShareConfigSchema` (all-false defaults) is only for normalizing what's *stored*, where a missing key must read as off.

**On the mobile side** ([src/components/groups/](apps/mobile/src/components/groups/), the `groups` tab and the root-stack drill-downs), that absence is what drives the UI: a field being `undefined` means *private*, never *zero*. Every slot a shared value could occupy renders either the value or a designed locked chip ([not-shared.tsx](apps/mobile/src/components/groups/not-shared.tsx)) — **never a blank**, which would read as "didn't log". Nothing substitutes a default for a withheld field. Share-config edits go through [shareConfigDiff](apps/mobile/src/lib/groups/index.ts) so only changed keys are sent. Leaderboard and roster copy always pairs the percentage with "logged n of m days", because unlogged past days score zero while the current week's remaining days are excluded — a bare percentage would read as judgement rather than a count.

`daily_summaries` is the read model behind all of this: one row per (user, local day), recomputed by [SummariesService](apps/api/src/summaries/summaries.service.ts) inside the same transaction as every diary write, snapshotting the `user_targets` row effective **for that day** — which is what stops a mid-week target change from rewriting past adherence. "Today" in a group read is each member's own local date per their `users.timezone`, never one global date.

That row has **two writers, and they must stay disjoint**: `recomputeDay` owns every nutrition column and never touches `weight_kg`; `recomputeDayWeight` owns `weight_kg` and never touches nutrition. They share a primary key, so merging their `ON CONFLICT ... SET` maps into one "tidier" upsert makes each writer clobber the other's column with a stale value ([writers.test.ts](apps/api/src/summaries/writers.test.ts) exists to catch that). The weight path deliberately snapshots **no targets** — a day with only a weigh-in must stay unscorable, or logging your weight would register as a 0 kcal adherence miss.

## Architecture

All paths below are inside `apps/mobile/`. Routing is file-based via expo-router (`main: "expo-router/entry"`). [src/app/_layout.tsx](apps/mobile/src/app/_layout.tsx) is the root layout: it loads the Space Grotesk / Instrument Sans fonts (`useFonts`, gating render on load), sets up the ThemeProvider and splash overlay, and gates first-run via a Stack — the `(onboarding)` group until `onboardingComplete`, then the `(tabs)` group plus the `add-food` / `food-detail` full-screen modals, `day-picker` (the calendar, a native `formSheet`), the Groups drill-downs (`group/[id]`, `member-day`, `create-group`, `join-group`, and `group-sharing` as a modal), and `profile`. Those push at the ROOT stack, above the tab group, so they carry their own [screen-header.tsx](apps/mobile/src/components/ui/screen-header.tsx) rather than the tabs' persistent `AppHeader`. `create-group`/`join-group` are deliberately pushes, not modals: each `replace`s itself with the group it produced, which would otherwise strand the group detail in a modal presentation.

Tab routes live in [src/app/(tabs)/](apps/mobile/src/app/(tabs)/): `index.tsx` (**Log — the landing tab**; it owns the index route because native tabs always open on `index.tsx` and there is no initial-tab override), `vitals.tsx` (the stat-tile grid — **only tiles with a real data source ship**: weight, plus protein and energy from today's diary. Steps/sleep/heart need Apple Health or Health Connect and water/fasting need their own log paths; rendering those with sample numbers would break the same promise the groups UI makes. Older draft cards remain unreferenced in [src/components/dashboard/](apps/mobile/src/components/dashboard/) — note `weight-card.tsx` there is superseded and fabricates its series from `sample-data.ts`, so don't wire it up), `groups.tsx`, and `recipes.tsx`. The group's [_layout.tsx](apps/mobile/src/app/(tabs)/_layout.tsx) renders the shared [app-header.tsx](apps/mobile/src/components/app-header.tsx) (plan icon · date · profile button) above the tabs — persistent across all tabs because native tabs don't provide a header.

**Profile & settings is not a tab.** [src/app/profile.tsx](apps/mobile/src/app/profile.tsx) pushes at the root stack from the `AppHeader`'s profile button, which is its only entry point — the tab bar is reserved for the surfaces opened daily, and profile is the app's lowest-frequency destination. Anything set-once (theme, notification preferences, data export) belongs behind that button too, not in the tab bar.

**Platform-split components** are a starter pattern: files with a `.web.tsx`/`.web.ts` suffix replace their native counterpart on web. Web is not a shipping target (see Project), so the `.web.*` files are effectively legacy — but keep them in sync if you touch the native side, since they still get bundled.

- [app-tabs.tsx](apps/mobile/src/components/app-tabs.tsx) uses `NativeTabs` from `expo-router/unstable-native-tabs` (native tab bar), while [app-tabs.web.tsx](apps/mobile/src/components/app-tabs.web.tsx) builds a custom floating header from the headless `expo-router/ui` Tabs primitives. **A new tab route must be registered in both files.**
- Same pattern for [animated-icon.tsx](apps/mobile/src/components/animated-icon.tsx)/[animated-icon.web.tsx](apps/mobile/src/components/animated-icon.web.tsx) (splash overlay) and [use-color-scheme.ts](apps/mobile/src/hooks/use-color-scheme.ts)/[use-color-scheme.web.ts](apps/mobile/src/hooks/use-color-scheme.web.ts).

**Theming — the "Kinetic" design system** lives in [src/theme/](apps/mobile/src/theme/): `palette.ts` (light/dark `ThemeColors` incl. role aliases `actionPrimary`/`inkStrong` and precomputed `*Soft` alphas), `typography.ts` (`Fonts` + the `Type` scale — Space Grotesk for display/ALL numerals with `tabular-nums`, Instrument Sans for UI/body), `tokens.ts` (`Spacing` 4px grid, `Radius`, `Motion`, `Elevation`), and `provider.tsx` (context `ThemeProvider` + `useTheme()` → `{ scheme, colors }`; also feeds the expo-router nav theme). The scheme resolves from `preference.ts` — a `system`/`light`/`dark` choice persisted through **MMKV specifically because it is synchronous**, so the first paint already has the right scheme instead of flashing light. `system` is stored as the choice itself, never as the scheme it currently resolves to, or "System" would freeze at whatever the OS was when it was picked. It is a **device** preference, not account data: it lives in `src/theme/`, not `src/store/`, and `clearLocalData` must never reset it — signing out or deleting the account leaves the phone looking how its owner set it. Never hardcode colors in screens — consume via `const { colors } = useTheme()`, `ThemedText`/`ThemedView`, or the kit in [src/components/ui/](apps/mobile/src/components/ui/) (Button, IconButton, Input, StatNumber, Badge, MacroBar, Card). **Role rules**: `accent` (lime) only for active states/progress/streaks/active tab (never body text, never lime text on light bg — use `accentText`); `macro*` colors only on macro visuals; `success`/`danger` only for status; selected/focused = 2px `focusRing` border. The provider's context values are frozen module constants — don't rebuild them per render (reactCompiler is on).

**Weight & the chart.** The weight vertical lives in [src/components/vitals/](apps/mobile/src/components/vitals/), the `weight/index` and `weight/history` root-stack routes, and [store/weight.ts](apps/mobile/src/store/weight.ts) (zustand + MMKV, generous cache — weight data is tiny and the tile must paint before any network call). [weight-chart.tsx](apps/mobile/src/components/vitals/weight-chart.tsx) is drawn with **react-native-svg**, chosen over Skia/victory-native because those add 4–6 MB to the download for one screen. Its gestures need the `GestureHandlerRootView` in the root layout — without it the scrub silently never fires on Android. The chart builds its own path and domain; the crosshair rides a Reanimated shared value on the UI thread and React only hears about a *change of selected point*, which is also what makes the haptic land once per data point. Do not move the gesture construction into `useMemo`: reactCompiler treats a `useMemo` body as render code and rejects shared-value writes inside it.

**The day switcher.** The Log tab shows any day: history back through the retained window, and up to **`DIARY_MAX_FUTURE_DAYS` (30) days ahead** for planning. A planned day is an ordinary `diary_entries` row with a future `entry_date` — there is no separate "planned" concept, so when the day arrives the plan simply *is* the log. [loggableEntryDateSchema](packages/shared/src/diary-schemas.ts) bounds writes (UTC today + 32, absorbing the timezone spread) because one entry dated 2099 would sit at the top of every `max(entry_date)` read forever; **reads stay unbounded** — a history query legitimately spans years.

The week strip ([day-strip.tsx](apps/mobile/src/components/log/day-strip.tsx)) lives on the Log tab, **not** in `AppHeader`, which renders above all five tabs where a selected day means nothing. The header keeps the streak pill, the date, and the way into the calendar ([day-picker.tsx](apps/mobile/src/app/day-picker.tsx), a root-stack `formSheet` — the OS owns the drag/detents/dismiss, so no bottom-sheet library is in the bundle). The calendar grid, the rings ([progress-ring.tsx](apps/mobile/src/components/ui/progress-ring.tsx), react-native-svg) and all date math ([lib/dates](apps/mobile/src/lib/dates/index.ts)) are hand-rolled: **no calendar or date library**, for the same size reasons as the weight chart, and because a date lib re-introduces exactly the local-`Date` DST bug class that `YYYY-MM-DD` string math avoids.

**Nothing about this is on the launch path.** The strip sums diary entries already in MMKV, so a cold start issues no request at all. `GET /v1/summaries/days` ([SummariesController](apps/api/src/summaries/summaries.controller.ts), ≤400 days — far wider than the diary's 31, because a summary row is ~120 bytes) is a **browse-time gap-filler** for days the device no longer holds, fetched when the calendar sheet opens. The rule between the two caches is one-directional and there is no merge: `daily_summaries` is a *projection* of diary entries, not an independent record, so **local entries win for every day they cover** and a summary is consulted only for days outside the window ([use-day-states.ts](apps/mobile/src/hooks/use-day-states.ts)). The one field that inverts is the **target** — a summary carries the target in force *for that day*, so a past day scores against its snapshot rather than today's goal.

Three invariants that keep the UI honest:
- **Absence is not zero.** `unknown` (not loaded) is distinct from `unlogged` (the day ended empty), which is distinct from a logged 0 — see [day-status.ts](apps/mobile/src/lib/diary/day-status.ts). `lastFullSync` on the diary store is what licenses reading an absent in-window day as empty; before it lands the cell renders a skeleton. Same promise the groups UI makes by never rendering a withheld value as a blank.
- **The streak is derived locally** (`localStreak`), so it is instant, offline, and never stale. The server's count is fetched *only* when the local run reaches the edge of the 90-day window and the real streak is longer — for most accounts, never. Future planned days can never inflate it; the walk goes backwards from today. Thresholds (`CALORIE_BAND` etc.) live in [packages/shared/src/summaries.ts](packages/shared/src/summaries.ts) so the client colours a day exactly as the server's group adherence scores it.
- **`pruneDays` is anchored on today** (`today−90 … today+31`, plus the day being viewed), never "the N latest keys". Once a day can be planned ahead, the future days are the chronologically largest and a `slice(-N)` evicts today and all history.

`checkDayRollover` follows midnight — via AppState for an app resumed the next morning, via Log tab focus for one left open overnight — and only moves the selection when the user was actually sitting on the old today.

**Units.** Every weight crossing the API is in kilograms; `unit` on a response is a display preference. Convert once at render via [lib/weight](apps/mobile/src/lib/weight/index.ts), never per point and never server-side — rounding in both places drifts the chart away from the history list. The server's 20–500 kg bound cannot catch a pound value typed as kilograms in the normal human range (165 lb entered as kg is a plausible 165 kg and is accepted), so the unit toggle in the log sheet has to stay visually unmissable.

**Path aliases** ([apps/mobile/tsconfig.json](apps/mobile/tsconfig.json), app-internal): `@/*` → `src/*`, `@/assets/*` → `assets/*`. Shared data shapes (catalog/food, health, diary, profile types) are imported from `@metabolizm/shared` — the app barrels (`@/lib/api`, `@/lib/food`, `@/lib/health`) export runtime helpers only, not types.

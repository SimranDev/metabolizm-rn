## F0 — BLOCKER — Route collision on `/weight` makes onboarding impossible to complete

**Area:** routing / onboarding
**Files:** `apps/mobile/src/app/(onboarding)/weight.tsx` vs `apps/mobile/src/app/weight/index.tsx`,
guard at `apps/mobile/src/app/_layout.tsx:94`

Two route files resolve to the same URL `/weight`:

- `(onboarding)/weight.tsx` — `(onboarding)` is a group, so it contributes nothing to the path
- `weight/index.tsx` — the weight-vertical drill-down

`weight/index` is registered **inside `<Stack.Protected guard={onboardingComplete}>`**
(`_layout.tsx:94`). During onboarding `onboardingComplete === false`, so the winning
`/weight` route is not mounted and every navigation to it is a silent no-op.

`height.tsx:34` does `router.push('/weight')`. Tapping **Continue** on "How tall are
you?" therefore does nothing at all: no navigation, no error, no JS exception. The
user is trapped on the height screen forever.

**Proof (deep links issued while in onboarding):**

| deep link | result |
|---|---|
| `metabolizmrn:///activity` | navigates → "How active are you?" |
| `metabolizmrn:///goal-weight` | navigates → "What's your goal weight?" |
| **`metabolizmrn:///weight`** | **no-op, screen unchanged** |

Sibling onboarding routes navigate fine, so this is specific to the colliding name.

**Impact:** no new user can finish onboarding on Android, so no new account can be
created through the UI at all. Note the emulator was already parked on "How tall are
you?" at the start of this session, which is consistent with someone hitting this and
giving up.

**Repro:** fresh install → Get started → goal → sex → dob → enter any valid height →
tap Continue → nothing happens.

## F1 — CRITICAL — Date of birth cannot be set on Android; age silently locked to 25

**Area:** onboarding / `dob` step
**File:** `apps/mobile/src/app/(onboarding)/dob.tsx:71`

The "When were you born?" screen renders **no date picker at all** on Android. The
screen shows the title, a large empty gap, a read-only "YOUR AGE / 25 yrs" readout,
and Continue. A UI-hierarchy dump of the entire gap region (y 480–1950) returns
**zero nodes** — the picker is not merely invisible, no view exists.

**Root cause (proven by experiment):** `styles.picker` is `{ alignItems: 'center' }`.
The Android `DateTimePicker` for `presentation="inline"` renders
`<Host matchContents={{ vertical: true }}>` — it reports an intrinsic height but
**no intrinsic width**. Under a cross-axis `center` parent it collapses to width 0
and Compose renders nothing.

Changing only that line to `{ alignSelf: 'stretch' }` made the picker appear
immediately (ComposeView at [72,624][1208,1302] with "Select date",
"Jul 20, 2001", and an editable `07/20/2001` field). **Change was reverted** —
reported, not fixed, per instruction.

**Impact:** every Android user's age defaults to 25 regardless of real age. Age is an
input to BMR/TDEE, so calorie and macro targets are wrong for every user who is not
25 — the core promise of the app. There is no error and nothing looks broken, so it
fails silently.

**Repro:** fresh install → Get started → any goal → any sex → observe the dob screen.

## F2 — HIGH (dev/QA) — React state-update warning on every cold start; its LogBox toast blocks the primary CTA

**Area:** app startup
**File:** `apps/mobile/src/store/profile.ts:36`

Every cold start logs:

> `Can't perform a React state update on a component that hasn't mounted yet. This
> indicates that you have a side-effect in your render function that asynchronously
> tries to update the component. Move this work to useEffect instead.`

Component stack: `<ContextNavigator />` (ExpoRoot.js:135) → `<ExpoRoot />` → `<App />`.

**Cause:** `onRehydrateStorage: () => (state) => state?._setHydrated()`. The profile
store is created at module scope and persists to **AsyncStorage** (async). When
rehydration resolves before React has mounted the tree, `_setHydrated()` calls
`set()` against an unmounted tree.

**Impact:**
- The warning is genuine (React anti-pattern), though production hides LogBox.
- In dev builds the LogBox toast is laid out at y=2567–2709 and **overlaps the
  "Get started" button** (y=2448–2604), and its touch target swallows taps on the
  CTA. Tapping "Get started" appears to do nothing until the toast is dismissed.
  This blocks manual QA and automated testing of the very first screen.

**Repro:** `adb shell pm clear com.anonymous.metabolizmrn` → launch → tap "Get started" → nothing happens.

## F3 — HIGH — Diary entries are never sent to the backend, but sign-up promises "sync across devices"

**Area:** diary / sync
**Files:** `apps/mobile/src/store/diary.ts:6`, `apps/mobile/src/lib/api/` (no diary client),
`apps/api/src/diary/diary.controller.ts:49`

Logged a real meal through the UI (Breakfast, "Chicken, ground, raw", qty 2 ×
4 oz crumbled). The app updated correctly — 323 cal, 39.3 g protein, 18.3 g fat,
0.1 g carbs, all arithmetically right. Then:

```
select count(*) from diary_entries;   -->  0
select count(*) from user_targets;    -->  0
```

Nothing reached the server. `apps/mobile/src/lib/api/` contains `catalog`, `groups`,
`users`, `weight` — **there is no diary client**, and `grep -rn diary apps/mobile/src/lib/api/`
returns nothing. `store/diary.ts:6` says it plainly: *"Persisted on-device via MMKV
as an interim before the backend."*

The backend side is **already built and waiting**: `PUT /v1/diary/entries`
(`diary.controller.ts:49` `upsertEntries`), plus `GET days`, `DELETE entries/:id`,
`GET recents`.

**Why this matters beyond "not wired up yet":**
1. `sign-up.tsx:81` tells the user *"Create an account to keep your plan and **sync
   across devices**."* That promise is not kept for the app's primary data.
2. `daily_summaries` is the read model behind **all group reads**. With no diary
   writes, nutrition rows are never recomputed, so every group member will show as
   never having logged — adherence %, streaks and leaderboards are all structurally
   zero. Groups cannot work correctly until diary sync lands.
3. Uninstall or device loss destroys all food history.

Treat as "known gap with under-appreciated blast radius" rather than a regression.

## F4 — MEDIUM — A new account's first session records the wrong timezone (entries can be filed to the previous day)

**Area:** users / timezone
**File:** `apps/mobile/src/app/_layout.tsx:62-64`

The root layout PATCHes `users.timezone` once per launch in a `useEffect(..., [])`,
fire-and-forget with `.catch(() => {})`. On the launch where a user *signs up*, that
effect has already run and failed (no session yet), so the account is created with
the server default `'UTC'` and stays that way until the **next** app launch.

Observed: emulator TZ `Pacific/Auckland` (UTC+12); newly created
`qa-20260721@example.com` had `timezone = UTC`. After a force-stop and relaunch it
self-corrected to `Pacific/Auckland`.

**Impact:** for the whole first session, every server-side "today" is computed in UTC.
At UTC+12 that means anything logged before noon local is filed to the **previous
day**; at UTC−7 anything logged after 5 pm lands on **tomorrow**. Streaks and
adherence for day 1 are wrong. Silent — nothing surfaces to the user.

Note this is currently masked by F3 (no diary reaches the server at all), so it will
only start biting once diary sync ships. Weight entries, which *do* sync, are exposed
to it today.

## F6 — HIGH — The goal weight from onboarding is never saved, so "Journey to goal" is dead for every user

**Area:** weight vertical
**Files:** `apps/mobile/src/store/weight.ts:208` (`setGoal`), `apps/mobile/src/app/weight/index.tsx:134`

Onboarding asks "What's your goal weight?" and validates it (25–400 kg). That value
lands in the local profile and **is never sent to the server**.

- The API client already has `putGoal()` → `PUT /v1/weight/goal`
  (`lib/api/weight.ts:87`), and the store wraps it as `setGoal`
  (`store/weight.ts:208`).
- `grep -rn "setGoal" apps/mobile/src` outside the store returns **nothing** —
  no screen, no onboarding step, no settings row ever calls it.
- There is also no other UI anywhere to set or edit a goal weight.

The weight detail screen reads the goal from the server (`index.tsx:65`
`data?.goal`), so for every user it is `null`. Confirmed on a fresh account that had
just entered a 75 kg goal in onboarding: `user_weight_goals` had no row, and the
screen rendered **GOAL "—"**, **TO GO "—"**.

**Impact:** the GOAL and TO GO chips are permanently empty, and the whole
"Journey to goal" progress block (`index.tsx:143-155`, incl. progress % and
"started <x>") never renders. A designed and implemented feature is unreachable
because one call is missing. The weight-projection maths that `compute.test.ts`
guards can never run against real user data.

## F7 — MEDIUM — Re-weighing on the same day appears to do nothing ("CURRENT" is really the morning weigh-in)

**Area:** weight vertical
**File:** `apps/mobile/src/app/weight/index.tsx:133`

Logged 74.5 kg, then logged again the same day as 165 lb (74.84 kg). Both rows
persisted correctly. But:

- `daily_summaries.weight_kg` stayed **74.50**
- the **CURRENT** chip stayed **164.2 lb** (= 74.5 kg)
- the new value *did* appear in History

This is **not** a data bug. `DAY_WEIGHT_RULE === "earliest"`: `pickDayWeightKg`
(`apps/api/src/weight/compute.ts:93`) deliberately keeps the day's *first* weigh-in,
and the server comment explains why — *"which is why the UI labels the value
**'morning weigh-in'** rather than 'your weight'."*

The mobile UI never implements that label. `grep -rni "morning" apps/mobile/src`
returns **nothing**; the chip is hardcoded `label="CURRENT"` (`index.tsx:133`).

**Impact:** a user who weighs again after breakfast sees their new number in History
while "CURRENT" refuses to move. That reads as a sync failure and invites a bug
report or repeated re-logging. The server already made the semantic decision; only
the label is missing.

## Groups — the masking invariant is CORRECT (verified end to end)

This is the app's most safety-critical behaviour and it holds. Test setup: added a
second member (`Dev`) to the group with `share_config` allowing **only** `streaks`,
and gave that user a `daily_summaries` row containing real numbers
(1850 kcal, 140 g protein, 3 meals, meal names, 81.2 kg).

`GET /v1/groups/:id/feed` returned for that member:

```json
{ "userId": "...", "name": "Dev", "date": "2026-07-21",
  "shared": { "streaks": true, "macros": false, ... },
  "logged": true, "streak": 1, "comments": [], "reactions": [] }
```

- `macros`, `calories`, `mealNames`, `mealDetail`, `weightTrend` are **absent
  keys — not `null`**, exactly as the allowlist rule requires.
- None of 1850 / 140 / 81.2 / the meal names appear anywhere in the payload.
- The requesting user's own card *does* include `macros`, so the masking is
  per-subject rather than blanket suppression.

`GET /v1/groups/:id/leaderboard` returned only `rank`, `daysLogged`,
`adherencePct`, `streak` — grepping the response for `energy|kcal|calorie|weight`
matched **nothing**.

In the UI, the withheld member renders "1-DAY STREAK", "Logged today" and the
explicit line **"Shares nothing beyond logging in this group"** — a designed locked
state, never a blank that could read as "didn't log".

Share defaults on creation were also correct: choosing **Friends** produced
`{macros: true, streaks: true, calories: false, mealNames: false, mealDetail: false,
weightTrend: false, adherenceOnly: false}`, matching `CATEGORY_SHARE_DEFAULTS`.

**One consequence worth flagging:** the group screen showed **"You — No logs yet
today"** immediately after I had logged a 323 kcal breakfast. That is F3 in action —
because the diary never reaches the server, `daily_summaries` has no nutrition row,
so the group correctly reports the user as not having logged. Groups will
under-report every member until diary sync ships.
